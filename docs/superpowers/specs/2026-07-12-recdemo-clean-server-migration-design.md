# RecDemo clean server migration design

Date: 2026-07-12
Status: user-approved design
Scope: l20 /data/Zijian/goal/RecDemo
Approved approach: full archive copy plus live-path in-place replacement

## 1. Objective

Replace the dirty l20 workspace at /data/Zijian/goal/RecDemo with a clean,
versioned, documented, reproducible RecDemo while preserving the complete old
workspace as /data/Zijian/goal/RecDemo_Past.

The migration must keep the path
/data/Zijian/goal/RecDemo/dataset/paper_raw_v1 available to the resident r5
and r6a controllers throughout the operation. It must not stop, signal,
restart, or reconfigure those controllers or any unrelated GPU process.

## 3. Selected migration architecture

The selected architecture has four phases:

~~~text
Phase A: immutable evidence capture
current RecDemo --full copy and verify--> RecDemo_Past

Phase B: clean staging construction
approved Git branch + four-domain frozen data + manuals + manifests
    --> RecDemo_Next_20260712

Phase C: staging verification
Git, imports, tests, data hashes, controller path resolution, disk gates

Phase D: live-path replacement
RecDemo_Next_20260712 --rsync into existing path--> RecDemo
~~~

The existing RecDemo directory path remains present during Phase D. This is
required because r5 and r6a queue manifests bind the exact dataset path.

Direct two-step rename and archive-backed dataset symlinks are rejected:

- A two-step rename introduces a path gap while resident controllers are live.
- A symlink from the new RecDemo into RecDemo_Past would make the archive an
  active runtime dependency and would defeat the archive boundary.

## 4. Archive contract

### 4.1 Archive target

The archive target is exactly:

~~~text
/data/Zijian/goal/RecDemo_Past
~~~

The migration must stop before copying if that path already exists.

### 4.2 Copy semantics

The old RecDemo is copied, not moved, before any cleanup. The copy must preserve
regular files, directories, symlinks, modes, mtimes, and Git metadata. It must
not use hard links because the archive and new workspace must not share mutable
inodes.

### 4.3 Archive evidence

The migration evidence root is:

~~~text
/data/Zijian/goal/recdemo_migration_20260712
~~~

It contains, outside RecDemo_Past:

- original_tree.tsv with relative path, type, bytes, and mtime;
- original_regular_files.sha256 with every old regular-file hash;
- archive_tree.tsv for RecDemo_Past;
- archive_verification.log;
- disk_before.txt and disk_after_archive.txt;
- process_before.json;
- git_before.txt;
- migration_events.jsonl.

RecDemo_Past remains an exact copy of the original source tree. No archive note
or generated manifest is inserted into RecDemo_Past after verification.

### 4.4 Archive acceptance

Archive acceptance requires all of the following:

1. source and archive file counts match;
2. source and archive total byte counts match;
3. every entry in original_regular_files.sha256 validates in RecDemo_Past;
4. a dry comparison reports no missing, extra, or content-different path;
5. RecDemo_Past is not referenced by any queue manifest or active process.

No cleanup of the live RecDemo is permitted before these conditions pass.

## 5. Clean source identity

### 5.1 Source branch

The clean repository source is:

~~~text
branch: codex/aaai27-seed100-controller
base revision: 0c085b8
~~~

Revision 0c085b8 is the audited base that records the completed E5 four-domain
baseline. The deployment revision is the exact 40-character HEAD produced by
the migration implementation commits. The deployment manifest must record that
SHA and require it to be a descendant of 0c085b8.

### 5.2 Transfer method

The server staging checkout is created from a Git bundle generated from the
local committed branch. This avoids relying on l20 GitHub credentials or
network availability. The bundle is hashed before transfer, hashed again on
l20, cloned into the staging directory, and checked out at the exact deployment
revision.

Untracked local tar files, temporary execution logs, and unrelated worktree
files are not included in the bundle.

### 5.3 Git cleanliness

The new RecDemo includes a tracked .gitignore covering:

- dataset;
- checkpoints and checkpoints-meta;
- log, logs, outputs, and runs;
- Python bytecode and cache directories;
- pytest caches;
- tar archives and temporary bundles;
- wget logs and NUL;
- server-specific runtime manifests.

Stable source, manuals, tests, ledgers, and role manifests remain tracked.
Server-specific data hashes and migration records live under
MANIFESTS/runtime and are ignored by Git. This preserves an empty
git status --short after deployment.

## 8. Required-file manifest

The tracked MANIFESTS/required_files.json uses this schema per entry:

~~~json
{
  "path": "single_train.py",
  "tier": "runtime",
  "role": "Hydra training and validation-selection entry point",
  "required": true,
  "runtime_consumers": ["PreferGrow host", "PreferGrow full"],
  "source_revision": "0123456789abcdef0123456789abcdef01234567",
  "sha256": "0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef",
  "size_bytes": 0
}
~~~

The hexadecimal values above illustrate field lengths only. The generated
manifest must contain the real deployment revision, real file SHA-256 values,
and positive or valid zero-byte sizes.

A human-readable MANIFESTS/required_files.tsv mirrors path, tier, role, size,
and hash. scripts/audit_recdemo_layout.py validates both representations,
required path existence, hash equality, unexpected runtime files, dataset
protocols, and external asset declarations.

## 10. Live-path replacement

### 10.1 Precondition gates

Before replacing any file in RecDemo:

- RecDemo_Past archive acceptance has passed;
- staging tests have passed;
- staging required-file audit has passed;
- all four dataset hashes match the old source;
- free space is at least 40 GiB;
- r5 and r6a PID plus Linux start tokens match the preflight snapshot;
- current queue manifests still resolve every RecDemo data path;
- no active process has an open writable file under RecDemo;
- the staging and target absolute paths resolve under /data/Zijian/goal.

### 10.2 Replacement semantics

The staging tree is synchronized into the existing RecDemo directory using an
archive-preserving, delete-aware copy. Because the four current dataset
directories were copied from the old tree with preserved size and mtime, they
remain content-identical during synchronization. Obsolete paths are removed
from RecDemo only after the verified RecDemo_Past archive exists.

The operation never enumerates paths in one shell and passes them to another
shell for deletion. All path resolution and mutation occur on l20 in one
controlled script. The script rejects empty, root, parent, symlinked, or
unexpected target paths.

### 10.3 Staging retention

RecDemo_Next_20260712 remains intact until all post-deployment checks pass. It
is removed only after verifying the deployed RecDemo and recording its tree
hash. If verification fails, staging remains available for repair and
RecDemo_Past remains available for full restoration.

## 12. Failure handling and rollback

The migration is fail-closed.

- If archive copy or archive verification fails, leave RecDemo untouched.
- If staging construction or tests fail, leave RecDemo untouched and preserve
  both staging and evidence logs.
- If disk falls below the threshold before replacement, stop before mutation.
- If controller identity changes during preflight, stop and re-audit.
- If live synchronization fails, do not delete staging. Repair by completing
  synchronization from staging or restore from RecDemo_Past.
- If post-deployment data hashes differ, restore the affected files from
  RecDemo_Past and mark the migration failed.
- RecDemo_Past is never deleted by the migration workflow.

A restoration uses archive-preserving synchronization from RecDemo_Past back
into RecDemo after resolving and verifying both absolute paths. Restoration is
not executed unless deployment verification fails or the user explicitly
requests it.

## 13. Resource estimate

Expected storage during migration:

- existing RecDemo: approximately 13 GiB;
- RecDemo_Past copy: approximately 13 GiB;
- staging clean repository plus four-domain data: below 1 GiB;
- evidence and manifests: small relative to model data.

Starting from approximately 77 GiB free, the archive phase is expected to leave
more than 60 GiB before old runtime content is removed from RecDemo. The hard
stop remains 40 GiB.

Expected wall time depends on disk throughput and full SHA verification.
Archive copy plus hashing is expected to dominate; the design allows it to run
without occupying a GPU.

## 14. Completion definition

The migration is complete only when all of these statements are evidenced:

1. /data/Zijian/goal/RecDemo_Past is a verified complete copy of the original
   dirty workspace;
2. /data/Zijian/goal/RecDemo is the clean deployment revision;
3. git status --short is empty;
4. the four current dataset directories and hashes are unchanged;
5. required-files and external-artifact manifests validate;
6. focused tests and compile checks pass;
7. the detailed operation manual exists and matches the deployed layout;
8. r5 and r6a were not interrupted;
9. no unrelated GPU process was modified;
10. post-deployment server status and disk evidence are recorded.

No old artifact is described as deleted. It is archived under RecDemo_Past or
left in its existing immutable external root.
